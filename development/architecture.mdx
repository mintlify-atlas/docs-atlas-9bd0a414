## System Architecture

CBDB Online Input System is built on **Laravel 12.x** with a traditional MVC architecture, enhanced with Repository pattern for data access abstraction.

## Backend Architecture

### MVC Pattern

The application follows Laravel's MVC pattern with clear separation of concerns:

```
app/
├── Http/
│   ├── Controllers/          # Request handling and response
│   │   ├── BasicInformationController.php
│   │   ├── CodesController.php
│   │   ├── OperationsController.php
│   │   └── Api/              # RESTful API endpoints
│   └── Middleware/           # Request filtering
├── Models/                   # Eloquent ORM models
│   ├── BiogMain.php
│   ├── User.php
│   └── Operation.php
├── Repositories/             # Data access layer
│   ├── BiogMainRepository.php
│   ├── OperationRepository.php
│   └── ToolsRepository.php
└── Services/                 # Business logic
    ├── NameSearchIndexService.php
    └── VariantCharNormalizer.php
```

### Repository Pattern

The system uses Repository pattern to abstract database operations:

<CodeGroup>
```php app/Repositories/BiogMainRepository.php
class BiogMainRepository {
    // Transaction-based office posting management
    public function officeStoreById($personId, array $data): array {
        return DB::transaction(function () use ($personId, $data) {
            // Lock person record
            $person = BiogMain::lockForUpdate()->find($personId);
            
            // Store office data
            $office = $this->storeOfficeData($data);
            
            // Update addresses
            $this->syncAddresses($office, $data['addresses'] ?? []);
            
            // Log operation
            $this->logOperation($personId, $office, 'create');
            
            return $office;
        });
    }
}
```

```php app/Repositories/OperationRepository.php
class OperationRepository {
    // Unified operation logging
    public function store(array $data): Operation {
        return Operation::create([
            'resource_type' => $data['resource_type'],
            'resource_id' => $data['resource_id'],
            'op_type' => $data['op_type'], // 1=create, 2=update, 3=restore, 4=delete
            'resource_data' => json_encode($data['resource_data']),
            'user_name' => auth()->user()->name ?? 'system',
        ]);
    }
}
```
</CodeGroup>

### Model Design

#### Single Primary Key Tables

Use **Eloquent ORM** for tables with single primary keys:

```php app/Models/BiogMain.php
class BiogMain extends Model {
    protected $table = 'BIOG_MAIN';
    protected $primaryKey = 'c_personid';
    public $timestamps = false;
    
    // Relationships
    public function altnames() {
        return $this->hasMany(AltnameData::class, 'c_personid');
    }
}
```

#### Composite Primary Key Tables

For tables with **composite primary keys**, use **Query Builder** directly:

```php
// ❌ Avoid: Eloquent doesn't support composite keys natively
// AltnameData model would have issues with delete()/update()

// ✅ Correct: Use DB facade for composite key tables
DB::table('ALTNAME_DATA')
    ->where('c_personid', $personId)
    ->where('c_sequence', $sequence)
    ->where('c_alt_name_chn', $name)
    ->where('c_alt_name_type_code', $typeCode)
    ->update(['c_notes' => $notes]);
```

<Info>
**Composite Key Tables**: `ALTNAME_DATA`, `POSTED_TO_ADDR_DATA`, `ASSOC_DATA`, `KIN_DATA`, `ENTRY_DATA`

These tables require Query Builder operations as Laravel Eloquent doesn't officially support composite primary keys.
</Info>

## Database Architecture

### Connection Strategy

The system supports multiple database environments:

```php config/database.php
'connections' => [
    'mysql' => [
        'driver' => 'mysql',
        'host' => env('DB_HOST', '127.0.0.1'),
        'database' => env('DB_DATABASE', 'cbdb'),
        'username' => env('DB_USERNAME', 'cbdb'),
        'password' => env('DB_PASSWORD', ''),
        'charset' => 'utf8mb4',
        'collation' => 'utf8mb4_general_ci',
    ],
    
    'sqlite' => [
        'driver' => 'sqlite',
        'database' => env('DB_DATABASE', ':memory:'),
        'foreign_key_constraints' => true,
    ],
],
```

### Schema Management

Database schema is managed through Laravel migrations:

- **Baseline migration**: `2025_01_01_000000_import_cbdb_schema.php` imports the complete CBDB schema
- **Incremental migrations**: All schema changes are tracked in separate migration files
- **Compatibility**: All migrations must work on both MySQL/MariaDB and SQLite

See [Database Schema](/development/database-schema) for detailed schema documentation.

The application uses Laravel's routing system:


## Authorization

The system implements role-based access control:

```php
// In controllers
public function restore(Operation $operation) {
    // Only active admins can restore
    if (!auth()->user()->is_admin || !auth()->user()->is_active) {
        abort(403, 'Unauthorized');
    }
    
    // Restore operation
    $this->operationRepository->restore($operation);
}
```

### User Roles

- **Regular User**: Basic data entry
- **Expert**: Advanced data management
- **Crowdsourcing**: Limited public contributions
- **Admin**: Full system access including operations restore

## Caching Strategy

Laravel's caching system is used for performance optimization:

```php
// Cache configuration codes
Cache::remember('codes.altname', 3600, function () {
    return DB::table('ALTNAME_CODES')->get();
});

// Clear cache on update
Cache::forget('codes.altname');
```
